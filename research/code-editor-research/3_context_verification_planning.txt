## Summary
- The highest leverage practice in Claude Code is to **give it verification criteria** (tests, screenshots, expected outputs).  
- Claude’s **context window** holds the entire conversation, files, and command outputs, but it fills quickly. Managing context is critical.  
- Without verification, the human becomes the only feedback loop. With verification, Claude can check its own work.  
- Recommended workflow has **four phases**:  
  - **Explore**: enter plan mode, Claude reads files and answers questions without changes.  
  - **Plan**: ask Claude to create a detailed plan, edit it if needed.  
  - **Implement**: switch to normal mode and code against the plan.  
  - **Commit**: open a PR or finalize changes. Skip planning for trivial fixes.  
- Prompts should be **precise and scoped**: describe symptoms, point to sources, reference existing patterns, and avoid vague instructions.  
- Use **rich content**: reference files with `@`, paste images, provide URLs, pipe command line data.  
- For larger features, let Claude **interview you** about implementation, edge cases, and trade‑offs before coding.  
- **claude.md** is loaded every session. Include only essential instructions (bash commands, style rules, testing steps, common gotchas). Remove anything Claude can infer from code to avoid bloat.  
- Quick recap:  
  - Manage the context window carefully.  
  - Provide verification criteria.  
  - Follow explore → plan → implement → commit workflow.  
  - Write specific, scoped prompts.  
  - Use rich content and interviews for complex features.  
  - Keep `claude.md` lean and focused.  

This defines the **core discipline for effective use of Claude Code**: verification, context management, structured workflow, and precise prompts.
